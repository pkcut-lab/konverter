# Dossier — pdf-passwort

## § 1 — Tool-Mechanik-Research

### Was das Tool tut

**pdf-passwort** deckt zwei komplementaere Operationen ab:

1. **Verschluesseln (Schuetzen):** Nutzer laedt ein ungesichertes PDF, vergibt ein
   Passwort (optional: Berechtigungseinschraenkungen), erhaelt ein AES-256-geschuetztes
   PDF als Download.
2. **Entschluesseln (Entsperren):** Nutzer laedt ein passwortgeschuetztes PDF, gibt
   das bekannte Passwort ein, erhaelt das entsperrte PDF als Download.

Beide Operationen werden als ein kombiniertes Tool ("PDF Passwort") mit zwei Tabs
geliefert. Dies entspricht dem Muster von pdf-aufteilen/pdf-zusammenfuehren.

### Technisches Verfahren

#### Variante A — Entschluesseln (client-seitig, sofort umsetzbar)

```
Input: passwortgeschuetztes PDF (File-Objekt) + Passwort (String)
Library: PDF.js (Mozilla, Apache-2.0)
Process:
  1. file.arrayBuffer() → Uint8Array
  2. pdfjsLib.getDocument({ data: uint8, password: userPassword })
  3. Wenn Passwort korrekt: pdfDoc.promise resolved
  4. Alle Seiten via pdfDoc.getPage(i) extrahieren
  5. Seiten auf OffscreenCanvas rendern → Blob-Array
  6. Alternativ (bevorzugt): pdf-lib laedt bereits entschluesselte Bytes
     und speichert unveraendert — nur der Schutz wird entfernt
  7. Uint8Array → Blob → URL.createObjectURL() → <a download>
Output: unveraendertes PDF ohne Passwortschutz
```

PDF.js unterstuetzt RC4-40, RC4-128, AES-128, AES-256 Entschluesselung client-seitig
[Quelle: 13]. Die Entschluesselungs-Operation erfordert das korrekte Passwort — kein
Brute-Force, kein Crack-Versuch.

#### Variante B — Verschluesseln (client-seitig, Einschraenkungen beachten)

```
Input: ungesichertes PDF + Passwort (String) + optionale Berechtigungen
Library-Status 2026-04-25:
  - pdf-lib (MIT): Verschluesselungs-Feature existiert (PR #917, PR #1015),
    aber Issues #1680, #1390, #1296 zeigen bekannte Bugs seit 2021.
    Produktiveinsatz mit Vorbehalt. [Quelle: 12]
  - qpdf.wasm: Vollstaendige PDF-Spec-Konformitaet inkl. AES-256, Owner+User
    Passwords — empfohlen fuer Produktion. Groesserer Bundle (~4MB).
  - WebCrypto (AES-GCM): Fuer proprietaere Verschluesselung verwendbar,
    aber NICHT PDF-Spec-konform — erzeugt kein Standard-PDF-Passwort.
    Nur fuer Custom-Container, nicht fuer interoperables PDF.

Empfehlung fuer kittokit: qpdf.wasm fuer Verschluesselung, PDF.js fuer
Entschluesselung. Beide MIT/Apache-2.0-kompatibel.
```

#### Passwort-Typen in PDF-Spec

| Typ | Zweck | Verbreitung |
|---|---|---|
| User-Passwort | Oeffnen des Dokuments | Standard — alle Tools |
| Owner-Passwort | Berechtigungen aendern | Erweitert — Sejda, PDF24, iLovePDF |

Owner-Passwort: Separate Eingabe, schraenkt Drucken/Kopieren/Bearbeiten ein. In
Basis-Version bewusst weglassen (YAGNI bis Phase-2-Feedback).

### Qualitaets- und Praezisions-Erwartungen

AES-256 (PDF 2.0 Spec, ISO 32000-2) gilt als aktueller Sicherheits-Standard. RC4
(PDF 1.3/1.4) ist deprecated und von NIST nicht mehr empfohlen [Quelle: 14]. Tool
MUSS AES-256 als Default verwenden, RC4 nicht anbieten.

## § 3 — Strengths / Weaknesses

### PDF24 Schuetzen + Entsperren [Quelle: 1, 2]

**Strengths:**
- Komplett kostenlos, kein Rate-Limit, kein Account
- Server in Deutschland — DACH-Nutzer mit DSGVO-Bedenken werden angesprochen
- AES-256 fuer Verschluesselung
- Granulare Permissions (7 Optionen: Drucken, Kopieren, Bearbeiten, etc.)

**Weaknesses:**
- Server-Upload bleibt Pflicht — trotz DE-Server kein Zero-Upload-Argument
- Werbung direkt im Tool-UI — "100% kostenlos dank Werbung"
- Zwei getrennte Tools fuer Schuetzen + Entsperren — schlechtere UX als ein
  kombiniertes Tool
- 1h Auto-Delete nicht auf 2h erhoehen trotz Konkurrenz-Standard

### iLovePDF Protect [Quelle: 3]

**Strengths:**
- Passwort-Staerke-Indikator mit expliziten Anforderungen (Upper/Lower, Zahl,
  Sonderzeichen, min. 8 Zeichen)
- Permissions-Granularitaet: Drucken, Aendern, Kopieren, Formulare, Assemblierung,
  Accessibility — differenzierte Kontrolle

**Weaknesses:**
- Hoechste Tracking-Dichte: 143 TCF-Vendors + 62 Ad-Partner
- Nur 2 Dateien/100MB im Free-Tier — schnell erschoepft
- Verschluesselungs-Algorithmus nicht kommuniziert — fehlendes Vertrauen-Signal
- Keine Entschluesselung im selben Tool

### Sejda Encrypt + Decrypt [Quelle: 5]

**Strengths:**
- Einziger Konkurrent der RC4-128, AES-128, AES-256 explizit zur Wahl stellt
- Ausfuehrlichste Permissions-Matrix: 7 Optionen inkl. "High resolution printing"
  separat von "Printing"
- Desktop-App fuer Offline-Nutzung

**Weaknesses:**
- RC4-Option ist Security-Theater: NIST deprecated, modernes PDF-Verstaendnis warnt
  aktiv davor — Sejda's Warnung "Some PDF readers will ignore these permissions" ist
  vertrauensuntergrabend
- Haerteste Free-Tier-Limits: 3 Tasks/h — bei Testnutzung schnell erschoepfend
- EN-only — DACH nicht adressiert

## § 5 — UX-Patterns (live beobachtet)

### Tab-Pattern vs. Zwei-Tool-Pattern

| Muster | Vertreter | Bewertung |
|---|---|---|
| Zwei separate Tools (Schuetzen + Entsperren) | PDF24, iLovePDF, Smallpdf | Nutzer muss zweimal navigieren; breitere SEO-Abdeckung |
| Ein Tool mit zwei Tabs/Modi | Sejda (auf einer Seite via Nav-Links), PDFCandy | Weniger Navigation; schlechtere separate Keyword-Rankings |

**Empfehlung:** Zwei separate Sub-URLs (`/pdf-passwort-hinzufuegen`, `/pdf-passwort-entfernen`), aber gemeinsamer `/pdf-passwort`-Hub mit interner Verlinkung. Erlaubt separate SEO-Rankings fuer beide Intents.

### Passwort-Eingabe-UX

- iLovePDF: Staerke-Indikator + explizite Anforderungs-Checkliste (Upper/Lower/Zahl/Sonderzeichen)
- PDF24: Einfaches Input ohne Staerke-Feedback
- Sejda: Zwei getrennte Felder (User-Password + Owner-Password) + Algorithmus-Dropdown

**Empfehlung:** Passwort-Staerke-Indikator wie iLovePDF — ohne es zu einer Pflicht
zu machen (Nutzer kennt seine Anforderungen). Bestaetigungsfeld (Passwort zweimal
eingeben) verhindert Tippfehler.

### Berechtigungs-UX

Alle Konkurrenten mit Permissions zeigen Checkboxen/Toggles. Pattern:
- "Drucken erlauben" / "Kopieren erlauben" / "Bearbeiten erlauben"
- Standardwert: Alle erlaubt (Opt-in-Einschraenkung, nicht Opt-out)

**Empfehlung:** Permissions als "Erweitert"-Accordion — ausgeblendet by default,
expandierbar. Reduziert kognitive Last fuer 80% der Nutzer.

### Privacy-Signaling (besonders kritisch fuer Passwort-Tool)

PDF-Passwort-Tools bearbeiten per Definition sensitive Dokumente. Privacy-Signaling
ist hier geschaeftskritisch:

- PDF24: "Server in Deutschland" + "1 Stunde" — stark
- Sejda: "Automatically deleted after 2 hours" — stark
- Adobe: Kein explizites Delete-Timing — schwach
- iLovePDF: Privacy-Statement trotz 143 TCF-Vendors — widerspruchlich

**Empfehlung fuer kittokit:** "Kein Upload — Ihr Dokument verlasst Ihren Browser nie"
als primaeres Differenzierungsmerkmal. Gilt fuer Entschluesselung sofort (client-side
via PDF.js), fuer Verschluesselung nach qpdf.wasm-Integration.

## § 7 — Content-Angle (Differenzierungs-Hypothese)

### Ueberstrapaziert in der Kategorie

- "Schnell und sicher" — bei allen Konkurrenten
- "Server in Deutschland" — PDF24 hat das Monopol, Kopieren ist wertlos
- "Keine Software noetig" — Standard-Aussage 2026, kein Differenzierungsmerkmal

### Unterrepraesantiert

- **"Kein Upload"** als explizites Privacy-Argument fuer sensible Dokumente:
  Kein Marktfuehrer communicates client-side processing auf der Hauptseite. Besonders
  stark fuer Passwort-Tools, da Nutzer wissen, dass das Dokument sensitiv ist.

- **Klare Erklaerung der Grenzen:** "Was schuetzt ein PDF-Passwort wirklich?" —
  Kein Konkurrent erklaert transparent, dass Owner-Password-Einschraenkungen von
  manchen PDF-Readern ignoriert werden (Sejda nennt es, aber im Kleingedruckten).
  Transparenz als Vertrauens-Aufbau.

- **"Kein Cracking" explizit:** Kein Konkurrent kommuniziert klar, dass das Passwort
  bekannt sein muss. Nutzer mit vergessenem Passwort sind fehlgeleiteter Traffic —
  fruehzeitig aufloesen verhindert Bounce.

### Empfohlenes Framing

"PDF-Passwortschutz der Ihrem Dokument gerecht wird — Verschluesselung direkt im
Browser, kein Upload, kein Server, kein Risiko. AES-256 Standard."

## § 8 — Edge-Cases

| Edge-Case | Verhalten | Nutzerentscheidung noetig |
|---|---|---|
| Falsches Passwort beim Entschluesseln | PDF.js wirft `PasswordException` → klare Meldung: "Falsches Passwort. Bitte erneut versuchen." | Nein (automatisch) |
| Vergessenes Passwort | Tool kann nicht helfen — Hinweis: "Passwort muss bekannt sein. Dieses Tool knackt keine Passwoerter." | Nein (Abweisungs-Hinweis) |
| Bereits ungesichertes PDF entschluesseln | PDF.js: kein Passwort noetig → Dokument wird direkt als Download ausgegeben + Hinweis: "Dieses PDF war bereits ungesichert" | Nein (automatisch) |
| Leeres Passwort beim Verschluesseln | Validation: Passwortfeld darf nicht leer sein. Pflicht-Validierung vor Submit. | Ja (Fehlermeldung) |
| Schwaches Passwort (1-3 Zeichen) | Warnung + Staerke-Indikator, aber kein Block — Nutzer entscheidet selbst | Nein (Warnung ohne Block) |
| PDF bereits verschluesselt + erneut verschluesseln | Zuerst entschluesseln (Tab 1), dann verschluesseln (Tab 2) — kein direktes Re-Encrypt | Ja (Workflow-Erklaerung) |
| Owner-Password-only (kein User-Password) | PDF.js unterstuetzt: leer-Password oeffnet das Dokument, Owner-Password sperrt Berechtigungen | Nein (transparenter Hinweis) |
| Beschaedigtes/Malformed PDF | Error-Handling: try/catch → "Diese Datei scheint beschaedigt zu sein" | Nein (Fehler-Feedback) |
| Sehr grosse verschluesselte PDFs (>100MB) | Client-side Memory-Belastung — ab 50MB Warnung anzeigen | Nein (Warnung) |

## § 9 — Differenzierungs-Hypothese (§2.4)

### A — Baseline-Features (was wir mindestens liefern muessen)

- Passwort-Entfernung mit bekanntem Passwort (Server-Upload-freie Alternative zu PDF24)
- Passwort-Hinzufuegen mit AES-256 (kein RC4)
- Kein Account, kein Rate-Limit
- Passwort-Staerke-Indikator (iLovePDF-Pattern)
- Automatische Datei-Beschreibung: Tool muss transparent machen, dass kein Passwort
  geknackt wird

### B — Differenzierungs-Features (was uns hebt)

1. **Client-Side-Processing** — "Kein Upload, kein Server, Ihr Dokument verlasst Ihren
   Browser nie." Kein einziger Top-7-Konkurrent bietet dies. Besonders stark fuer
   Passwort-Tools, da Dokument per Definition sensitiv. Basis: PDF.js fuer
   Entschluesselung (sofort umsetzbar), qpdf.wasm fuer Verschluesselung.

2. **Ein kombiniertes Tool** — Schutzen + Entsperren in einem Interface (zwei Tabs),
   nicht zwei separate Seiten wie bei PDF24. Nutzerfreundlicher Workflow fuer den
   Anwendungsfall "Ich muss das Passwort wechseln" (Tab 1: Entsperren → Tab 2: neu
   Schuetzen).

3. **Transparente Security-Kommunikation** — Erklaerungstext: was AES-256 bedeutet,
   was Owner-Password vs. User-Password ist, und explizit: "Passwoerter koennen nicht
   wiederhergestellt werden — bitte sicher aufbewahren." Kein Konkurrent macht das
   klar und zugaenglich.

### C — Bewusste Luecken (was wir NICHT bauen und warum)

- **Kein Passwort-Knacken (Brute-Force/Dictionary):** Verletzt Privacy-First-Prinzip,
  rechtlich risikoreich, YAGNI da nicht in Scope.
- **Keine Certificate-Based-Encryption (Digital Signatures):** Zu komplex fuer Basis-
  Anwendungsfall, kaum Nachfrage im Consumer-Segment.
- **Keine Cloud-Storage-Integration (Drive/Dropbox):** Widerspricht Client-Side-USP
  (Upload-Abhangigkeit).
- **Kein Batch-Encrypt mehrerer Dateien gleichzeitig:** §4 Pain zeigt Single-Document-
  Focus bei Nutzer-Bedarf; YAGNI bis Phase-2-Feedback.
- **Keine RC4-Option:** Deprecated, sicherheitstechnisch nicht vertretbar, trotz
  Sejda-Precedenz.

### D — Re-Evaluation-Trigger

- Phase-2-Analytics: CTR / Bounce nach 60 Tagen pruefen — if Bounce > +20%,
  Pain-Point-Revision
- User-Feedback: ≥3 Reports zu "Passwort vergessen"-Anfragen → ggf. Erklaer-Content
  ausbauen oder auf qpdf.wasm-Recovery-Feature pruefen (rechlich klaeren)
- Competitor-Trigger: Wenn PDF24 oder iLovePDF Client-Side-Verschluesselung launcht →
  sofortiger Full-Refresh + H1 re-evaluieren
- qpdf.wasm-Reife: Wenn qpdf.wasm-Bundle stabiler und kleiner wird → Verschluesselungs-
  Qualitaet upgraden ohne UX-Aenderung

## § 11 — Security-Guidance (Tool-Type: Formatter + Keywords: password, crypto, encrypt)

**Trigger:** Tool-Type `Formatter` + Keyword-Match (`password`, `crypto`, `encrypt`,
`hash`, `random`) → Security-Research Pflicht nach AGENTS.md §Schritt-D2.

### Threat-Model

Passwort-basierte PDF-Verschluesselung erfordert:
(a) kryptografisch sicheres Zufalls-IV/Salt (kein `Math.random()`),
(b) AES-256-CBC oder AES-256-GCM via Standard-PDF-Encryption (nicht proprietaere
WebCrypto-Only-Verschluesselung, die kein Standard-PDF-Reader oeffnen kann).
Haupt-Risiken: schwacher PRNG fuehrt zu vorhersagbaren IVs → Angriffs-Vektor;
RC4 ist gebrochen; proprietaere Verschluesselung erzeugt inoperables PDF.

### Pflicht-Patterns

```javascript
// IV/Salt-Generierung — PFLICHT CSPRNG
const iv = crypto.getRandomValues(new Uint8Array(16)); // fuer AES-CBC
const iv = crypto.getRandomValues(new Uint8Array(12)); // fuer AES-GCM

// Passwort → Key-Derivation via PBKDF2 (wenn WebCrypto verwendet wird)
const keyMaterial = await crypto.subtle.importKey(
  "raw", new TextEncoder().encode(password), "PBKDF2", false, ["deriveBits", "deriveKey"]
);
const key = await crypto.subtle.deriveKey(
  { name: "PBKDF2", salt: crypto.getRandomValues(new Uint8Array(16)),
    iterations: 100000, hash: "SHA-256" },
  keyMaterial, { name: "AES-GCM", length: 256 }, false, ["encrypt"]
);

// Fuer Standard-PDF-Encryption: qpdf.wasm verwenden, nicht WebCrypto direkt
// WebCrypto erzeugt kein PDF-Spec-konformes Encrypted-PDF
```

### Verboten

```javascript
Math.random()           // verboten fuer IV, Salt, oder jede sicherheitsrelevante Zufallszahl
RC4                     // verboten — NIST deprecated, PDF-Spec 1.3/1.4 Legacy
AES-128                 // nicht empfohlen — AES-256 ist Standard
eval(userInput)         // verboten
// Proprietaere Verschluesselung die kein Standard-PDF erzeugt — verboten
```

### Quellen
