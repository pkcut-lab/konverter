# Dossier -- kgv-rechner

## 1 -- Tool-Mechanik-Research

### Formel-Varianten

**Variante 1 -- Basis-KGV (2 Eingaben, Standard):**
```
KGV = Aktienkurs / Gewinn je Aktie (EPS)
```
Oder aequivalent:
```
KGV = Marktkapitalisierung / Jahresueberschuss
```
Alle deutschen Konkurrenten verwenden Variante 1 als Primaerformel [Quellen: 1, 3, 4, 5, 6].

**Variante 2 -- Gewinnrendite (inverse Ausgabe):**
```
Gewinnrendite (%) = (1 / KGV) x 100
```
Beispiel: KGV 20 -> Gewinnrendite 5 %. Ermoeglicht direkten Vergleich mit Anleiherenditen.
Keiner der gefundenen deutschen Konkurrenten berechnet die Gewinnrendite automatisch [Quelle: 9].

**Variante 3 -- Forward-KGV (prognostizierter EPS):**
```
Forward-KGV = Aktienkurs / prognostizierter EPS (naechste 12 Monate)
```
Bezeichnet mit "e" (z.B. KGV 2025e). Nutzbar wenn Analysten-Schaetzungen bekannt [Quelle: 7].

**Variante 4 -- Trailing-KGV (TTM):**
```
Trailing-KGV = Aktienkurs / EPS der letzten 12 Monate (TTM)
```
Bezieht sich auf tatsaechliche Gewinne, nicht Prognosen. Stabiler, aber vergangenheitsorientiert.

**Variante 5 -- Shiller-KGV / CAPE:**
```
Shiller-KGV = Aktienkurs / Durchschnitt inflationsbereinigter EPS der letzten 10 Jahre
```
Glaettet konjunkturelle Schwankungen. S&P 500 langfristiger Median ca. 17 [Quelle: 8].
Relevant als Bildungs-Content im Artikel-Block; nicht als direkte Berechnungsfunktion.

**Variante 6 -- PEG-Ratio (Erweiterung):**
```
PEG = KGV / erwartetes Gewinnwachstum (%)
```
PEG < 1 = potentiell unterbewertet; PEG > 1 = teuer relativ zum Wachstum [Quellen: 9, 11].
Kein DE-Konkurrent bietet PEG im selben Tool; eigenes Tool in Masterplan-Pipeline geplant.

|---|---|---|

| KGV < 5 | Extrem niedrig | Warn-Badge (non-blocking): "Moegliche Value-Trap -- Gewinne koennten fallen" |

## 4 -- User-Pain-Points

Quellen: HN Algolia-API [12], aggregierte DE-Finanzforum-Threads [7, 10, 11], allgemeine Stock-Calculator-Feedback-Analyse [13].

**Pain-Point 1 -- Kontext fehlt komplett (Quality/UX):**
> "Das KGV allein sagt mir nichts -- ich weiss nicht ob 18 gut oder schlecht ist fuer meine Branche."

Alle Konkurrenten liefern eine nackte Zahl ohne Einordnung. Nutzer verstehen nicht ob ihr KGV
ueber- oder unterdurchschnittlich fuer den Sektor ist. Fuehrt zu Frustration und erneuter Suche. [Quellen: 7, 10, 11]

**Pain-Point 2 -- Negatives KGV verwirrend (Quality):**
> "Das Tool zeigt -15 an wenn das Unternehmen Verluste macht. Was bedeutet das?"

Keiner der Konkurrenten behandelt negatives EPS sauber. Das negative KGV-Ergebnis wird ohne
Hinweis angezeigt, obwohl es inhaltlich nicht interpretierbar ist. [Quelle: 10, 11]

**Pain-Point 3 -- Forward vs. Trailing Verwirrung (UX):**
> "Welchen Gewinn soll ich eingeben? Den vom letzten Jahr oder die Prognose?"

Alle Tools haben genau ein EPS-Feld ohne Erklaerung ob Trailing oder Forward gemeint ist.
Analysten-Schaetzungen vs. Ist-Zahlen fuehren zu verschiedenen KGVs -- Nutzer wissen nicht
welche Zahl sie nehmen sollen. [Quellen: 6, 7, 9]

**Pain-Point 4 -- Keine Verknuepfung zum PEG (Missing Feature):**
> "Das KGV von 40 ist hoch, aber das Unternehmen waechst mit 30% p.a. -- da ist das doch eigentlich guenstig."

Fortgeschrittene Nutzer verstehen: KGV allein reicht nicht. Sie brauchen PEG oder zumindest einen
Hinweis auf die Wachstumsrate. Kein DE-Konkurrent bietet das. [Quelle: 12]

**Pain-Point 5 -- Woher bekomme ich den EPS? (UX):**
> "Ich weiss nicht wo ich den Gewinn je Aktie herbekomme."

Einsteiger-Nutzer scheitern schon an der Datenbeschaffung. Tools erwaehnen "Geschaeftsbericht" oder
"Boersenportal" vage -- ohne konkrete Anleitung wo man EPS-Daten in DE findet. [Quelle: 1]

Alle Konkurrenten loesen nur das Berechnungs-Problem (2 Zahlen -> 1 Ergebnis), nicht das

1. Gewinnrendite als Gegenstueck

Das macht das Tool fuer Einsteiger sofort nutzbar ohne zusaetzliches Recherchieren.

## 9 -- Differenzierungs-Hypothese (fuer 2.4 des Build-Tickets)

### A. Baseline-Features (Pflicht -- was jeder Konkurrent hat)

1. Eingabe: Aktienkurs + Gewinn je Aktie (beide in EUR, DE-Komma-normalisiert)
2. Ausgabe: KGV-Wert (2 NKS)
3. Formel sichtbar: "KGV = Aktienkurs div Gewinn je Aktie"
4. Edge-Case: Negatives EPS -> klarer Hinweis statt negativem Zahlenwert
5. Kurze Erklaerung was KGV bedeutet (max. 3 Saetze im Tool-Block)
6. Responsives Layout, Mobile-First

### B. Differenzierungs-Features (Was uns hebt)

1. **Gewinnrendite als automatische Zusatz-Ausgabe** -- "weil kein Konkurrent das berechnet + direkter Anleihe-Vergleich-Wert"
   - Ausgabe: Gewinnrendite = 1 / KGV x 100, z.B. "4,76 %"
   - Erklaerung: "entspricht einem Zins von X % auf den Kaufpreis"

2. **Historischer Kontext-Badge** -- "weil alle Nutzer fragen ob das KGV gut oder schlecht ist"
   - Ampel vs. DAX-Historisch (~14,6) und S&P-Historisch (~17,5)
   - Gruen: KGV < 15; Gelb: 15-22; Rot: > 22 -- mit Disclaimer "Branchenkontext beachten"
   - Statisch eingebaut, kein API-Call noetig

3. **Sektor-Benchmark-Dropdown (optional, Phase 2)** -- "weil Pain-Point Nr. 1 der haeufigste: kein Branchenkontext"
   - Auswahl: Automobil / Banken / Versorger / Konsumgueter / Tech / Industrie / Keine Angabe
   - Zeigt Sektor-KGV-Band und ob Nutzer-KGV darunter oder darueber liegt
   - Rein statisch (kein API), vollstaendig client-side

4. **EPS-Quelle-Hilfe** -- "weil Pain-Point Nr. 5: Einsteiger wissen nicht woher EPS-Daten kommen"
   - Tooltip oder Info-Icon: "Wo finde ich Gewinn je Aktie? Jahresbericht oder kostenlose Portale wie finanzen.net oder onvista.de"
   - Text-Only, keine externen Links als Interaktionspunkte

### C. Bewusste Luecken

- **Kein Echtzeit-Aktienkurs-Pull**: Erfordert externe API -- verletzt Non-Negotiable "kein Server-Upload"
- **Kein Shiller-KGV im Rechner**: Braucht 10 Jahre EPS-Daten -> Datenbeschaffung unrealistisch fuer Nutzer; bleibt Bildungs-Content im Artikel
- **Kein PEG-Rechner im selben Tool**: Scope-Creep; PEG ist in der Masterplan-Pipeline als eigenes Tool vorgesehen
- **Kein Multi-Aktien-Vergleich**: Tabellen-UI wuerde Komplexitaet stark erhoehen; Einzelbewertung genuegt fuer 90% der Faelle
- **Kein CSV-Export**: Keine Evidenz fuer Bulk-Anwendungsfaelle

### D. Re-Evaluation-Trigger

- Phase 2 Analytics: Wenn Sektor-Dropdown < 10 % Nutzung -> Feature aus Phase 2 streichen
- Analytics-Dip: Bounce > +20 % oder CTR < -15 % ueber 14 Tage -> sofortige Section-2/4-Aktualisierung
- Competitor-Launch: Falls Finanzfluss oder WiWo Sektor-Benchmark-Vergleich einfuehrt -> Differenzierungs-Prio B.3 hochstufen
- User-Feedback >= 3 gleiche Requests -> Sofort-Refresh Section 4

## 11 -- Security-Guidance (tool-type-abhaengig)

**Tool-Type:** Calculator -- kein Generator, kein Regex-Input vom Nutzer.

- **Threat-Model:** Nutzer gibt numerische Werte ein; Division by Zero und negatives EPS sind die einzigen mathematischen Edge-Cases. Kein ReDoS-Risiko. Kein XSS-Risiko da keine dynamische HTML-Insertion aus User-Input.
- **Pflicht-Pattern:**
  ```javascript
  // Input-Normalisierung DE (Tausenderpunkt strippen, Komma -> Punkt)
  function parseDE(val) {
    return parseFloat(val.replace(/\./g, '').replace(',', '.'));
  }
  // Division-by-Zero Guard
  if (eps === 0) return { error: 'EPS darf nicht 0 sein' };
  // Negative EPS Guard
  if (eps < 0) return { error: 'negativ', kgv: null };
  // Kurse <= 0 Guard
  if (kurs <= 0) return { error: 'Kurs muss > 0 sein' };
  ```
- **Verboten:** `eval(userInput)`, `innerHTML = userInput` (kein direktes HTML-Inject aus Formularfeldern)
- **Quellen:** OWASP Input Validation (https://owasp.org/www-community/vulnerabilities/Improper_Input_Validation), MDN parseFloat
