# Dossier — pdf-zusammenfuehren

## § 1 — Tool-Mechanik-Research

### Was das Tool tut

**pdf-zusammenfuehren** kombiniert zwei oder mehr PDF-Dateien zu einem einzigen zusammenhaengenden PDF-Dokument. Der Nutzer laedt mehrere PDFs, legt deren Reihenfolge fest, und erhaelt ein zusammengefuehrtes PDF als Download — alles client-seitig ohne Server-Upload.

### Technisches Verfahren (client-side, pdf-lib MIT)

```
Input: N × PDF-Datei (File-Objekt via <input multiple> oder Drag-and-Drop)
Library: pdf-lib (MIT, ~1MB JS-Bundle)
Process:
  1. Fuer jede Eingabedatei: file.arrayBuffer() → PDFDocument.load(buffer)
  2. Ziel-Dokument erstellen: PDFDocument.create()
  3. Fuer jede Quell-PDFDocument: copiedPages = await dest.copyPages(src, src.getPageIndices())
  4. copiedPages.forEach(p => dest.addPage(p))
  5. dest.save() → Uint8Array → Blob → URL.createObjectURL() → <a download> trigger
Output: einzelnes PDF (Dateiname: "zusammengefuehrt.pdf" oder aus erstem Dateinamen)
```

**Wichtiger Mechanismus:** `PDFDocument.copyPages()` kopiert Seiten inklusive aller Einbettungen (Schriften, Bilder, Formular-Felder, Annotations). Es werden keine Inhalte gerendert oder neu enkodiert — der Merge ist verlustfrei und schnell [Quelle: 8, pdf-lib README].

### Technische Edge-Cases

| Input-Klasse | Verhalten | Nutzerentscheidung noetig |
|---|---|---|
| Passwortgeschuetztes PDF | PDFDocument.load() wirft `EncryptedPDFError` — expliziter Passwort-Prompt noetig | Ja: Input-Feld "Passwort fuer Datei X" |
| Leer-PDF (0 Seiten) | copyPages gibt leeres Array → keine Seite wird hinzugefuegt → Warnung | Nein (automatisch: Datei wird uebersprungen + Hinweis) |
| Sehr grosse Dateien (>200 MB) | Hoeher RAM-Bedarf im Browser — moderne Desktop-Browser halten 1+ GB, Mobil kritisch | Ja: Warnung ab konfiguriertem Schwellenwert |
| Viele kleine Dateien (>50 Dateien) | Zahl der ArrayBuffer-Ladungen erhoeht Ladezeit linear | Nein (kein technischer Blocker) |
| Beschaedigtes/Malformed PDF | PDFDocument.load() wirft Exception — try/catch zeigt Fehler pro Datei | Nein (Fehler-Feedback pro Datei) |
| Gemischte Seitengroessen (A4 + Letter) | pdf-lib preserviert original Seitengroessen — kein Resizing | Nein (bewusste Entscheidung: Seiten unveraendert) |
| Doppelte Dateien | Technisch kein Problem — Nutzer sieht Datei zweimal in Liste | Nein (erlaubt, kein Block) |
| Nicht-PDF-Datei (.docx mit .pdf umbenannt) | `%PDF-` Magic-Bytes fehlen → Validation-Fehler vor Load | Nein (pre-load Validation) |

### Praeziions-Erwartungen und Qualitaet

PDF-Merge via pdf-lib ist **verlustfreies Byte-fuer-Byte-Kopieren der PDF-Seitenstruktur**. Qualitaetsverlust ist strukturell ausgeschlossen — es wird nicht gerendert oder reenkodiert. Einzige Ausnahme: Wenn das Quell-PDF bereits niedrige Qualitaet hat, aendert sich das nicht.

**Metadaten-Verhalten nach Merge:**
- Autor, Titel, Erstellungsdatum aus Quell-PDFs bleiben im Output erhalten (pdf-lib schreibt neue Creation-Metadata) [Quelle: 8]
- Wer Metadaten kontrollieren will: `dest.setAuthor('')` etc. — kein Konkurrent bietet GUI hierfuer

### Sprach-Varianten

DE-Markt verwendet: "PDF zusammenfuehren", "PDF zusammenfuegen", "PDFs verbinden", "mehrere PDFs zu einem zusammenfassen", "PDF joinen" (Anglizismus, seltener). In SEO-Headlines: "PDF zusammenfuehren" (mit Umlaut) und "PDF zusammenfuehren" (ohne) sind beide relevant.

## § 8 — Edge-Cases + Validation-Regeln

### Allowed

- Mehrere PDFs (2 bis n, nur RAM-begrenzt)
- Leere Seitenpuffer in Datei (PDF mit 0 Seiten erzeugt Warnung, wird uebersprungen)
- Gemischte Seitengroessen (A4 + Letter + A5 — alle werden unveraendert uebernommen)
- Gleichnamige Dateien (technisch kein Problem — Duplikate erlaubt)
- Grosse Dateien: Warnung ab 100 MB Gesamt-RAM (configurable threshold)

### Blocked / Validation-Pflicht

| Input-Klasse | Validierungsregel | User-Message (DE) |
|---|---|---|
| Nicht-PDF-Datei (kein `%PDF-` Magic Header) | Verboten | "Diese Datei ist kein gueltiges PDF: [Dateiname]" |
| Passwortgeschuetztes PDF | Passwort-Prompt (nicht geblockt) | "Bitte das Passwort fuer [Dateiname] eingeben." |
| Beschaedigtes PDF (pdf-lib wirft Exception) | Datei wird uebersprungen | "Datei [Dateiname] ist beschaedigt und wird uebersprungen." |
| Leeres PDF (0 Seiten) | Warnung, wird uebersprungen | "[Dateiname] enthaelt keine Seiten — wird uebersprungen." |
| Nur 1 Datei ausgewaehlt | Block: Merge benoetigt min. 2 Dateien | "Bitte mindestens 2 PDF-Dateien auswaehlen." |
| RAM-Warnung (Gesamt > 100 MB) | Warnung (kein Block) | "Grosse Dateien erkannt. Die Verarbeitung kann etwas dauern." |
| Nicht-PDF-MIME-Typ (accept-Filter wird umgangen) | Magic-Bytes-Validation vor Load | "Nur PDF-Dateien sind erlaubt." |

### Boundary-Tests

- 2 Dateien, je 1 Seite → OK
- 50 Dateien, je 10 Seiten → OK (500 Seiten gesamt, Browser haelt das)
- 1 Datei 0 Seiten, 1 Datei 5 Seiten → Warnung + 5 Seiten Output
- Alle Dateien leer → Fehler "Keine Seiten vorhanden"
- Mixed: 1 gueltiges PDF + 1 beschaedigtes → Beschaedigtes uebersprungen, gueltiges gemerged

## § 9 — Differenzierungs-Hypothesen

### H1 — Client-Side (Kein Upload) im DE-Markt einzigartig

**These:** Kittokit ist der einzige deutschsprachige PDF-Merger, der vollstaendig client-seitig verarbeitet — Dateien verlassen den Browser nie. Kein Konkurrent (iLovePDF, Smallpdf, PDF24, Adobe, online2pdf, Sejda) bietet das auf Deutsch.

**Beleg:** §2 Matrix: 6 von 7 Konkurrenten sind server-upload-basiert. pdfux.com ist der einzige WASM-Anbieter, hat aber keine DE-SEO-Praesenz und keine DE-UI. §4 Pain 1+2+3: Privacy und Upload-Aversion sind die staerksten ungeloesten Pain-Points. FBI-Warnung Maerz 2025 hat Nutzer-Sensibilitaet massiv erhoecht.

**Wert-Beleg:** `pdf zusammenfuehren ohne upload` ist ein Keyword-Gap (§6) — transactionaler Intent, kaum DE-Konkurrenz. Direkte Ranking-Chance.

---

### H2 — Keine Limits als explizites Versprechen

**These:** Waehrend alle Konkurrenten kuenstliche Limits haben (Sejda: 3 Tasks/h, iLovePDF: 25 Dateien/100 MB, Sejda: 50 Seiten), kann kittokit "so viele Dateien wie RAM erlaubt" anbieten — und das transparent kommunizieren. Das adressiert direkt §4 Pain 4 (Paywall-Pain) und unterscheidet sich von Sejda/iLovePDF strukturell.

**Beleg:** §2 Matrix, Spalte "Datei-Limits (Free)": alle Upload-basierten Dienste sind technisch zur Limitierung gezwungen (Server-Kosten). Client-seitig entfaellt dieser Zwang. §4 Pain 4: "usually hit you with a 'File too large' error or a paywall after two uses" ist ein dokumentierter Schmerzpunkt.

**Wert-Beleg:** "Keine Limits" als Hero-Copy auf einer client-seitigen Tool-Seite ist glaubwuerdig und pruefbar (kein Paywall-Bait) — starker Vertrauensanker.

---

### H3 — Metadaten-Transparenz nach Merge

**These:** Kein einziger untersuchter Konkurrent erklaert oder gibt dem Nutzer Kontrolle darueber, welche Metadaten (Autor, Erstellungsdatum, Software-Version) nach dem Merge im Output-PDF erhalten bleiben. Kittokit kann (a) dieses Thema im Content aufklaeren und (b) eine optionale "Metadaten bereinigen"-Checkbox anbieten.

**Beleg:** §4 Metadata-Pain: "creation timestamps, software versions, and author info can persist through merges unless explicitly stripped" — kein Konkurrent loest dieses Problem. §1: pdf-lib unterstuetzt `.setAuthor()`, `.setTitle()`, `.setCreationDate()` auf PDFDocument-Ebene — technisch trivial umzusetzen.

**Wert-Beleg:** Information-Gain fuer SEO (§7), und Privacy-Mehrwert fuer Nutzer, die Dokumente aus verschiedenen Quellen zusammenfuehren (Vertragsseiten, Formulare aus verschiedenen Abteilungen).

## § 11 — Security-Guidance (tool-type: formatter)

**Trigger:** tool_type = `formatter` — Pflicht-Section gemaess AGENTS.md §2 Schritt D2.

**Threat-Model:** Nutzer laedt beschaedigtes oder boesartiges PDF hoch. Der Client-seitige pdf-lib-Parser kann bei malformierten PDF-Bytesequenzen (Zip-Bombs im PDF-Content-Stream, eingebettete Makros, korrupte XRef-Tabellen) Exceptions werfen oder im schlimmsten Fall den Browser-Tab einfrieren. Da alles client-seitig laeuft, gibt es kein Server-Exploit-Risiko — die Angriffsoberflaeche beschraenkt sich auf den Browser-Tab und den Local-File-Output.

**Pflicht-Pattern (try/catch um jeden PDF-Load):**

```typescript
// PII-sicher: Dateiname wird nur lokal angezeigt, nie geloggt oder hochgeladen
async function loadPDF(file: File): Promise<PDFDocument | null> {
  try {
    // Magic-Bytes-Check BEVOR pdf-lib aufgerufen wird
    const header = await file.slice(0, 5).arrayBuffer()
    const magic = new TextDecoder().decode(header)
    if (!magic.startsWith('%PDF-')) {
      throw new Error('Kein gueltiges PDF')
    }
    const buffer = await file.arrayBuffer()
    return await PDFDocument.load(buffer)
  } catch (err) {
    // Kein Stack-Trace zum Nutzer — nur freundliche Fehlermeldung
    return null // Caller zeigt Fehler per Dateiname
  }
}
```

**Verboten:**
- `PDFDocument.load(buffer, { ignoreEncryption: true })` ohne Passwort-Prompt — wuerde verschluesselte PDFs ohne Erlaubnis oeffnen
- Dateiinhalt (Bytes oder Text) ins DOM schreiben ohne Sanitization (XSS via PDF-Metadaten moeglich wenn Autorname ungefiltert in `innerHTML` geschrieben wird)
- `URL.createObjectURL()` Blob ohne Revoke nach Download — Memory-Leak

**Empfehlung Groessen-Limit:**
- Client-seitig kein hard-limit noetig (Server-Kosten entfallen), aber: bei Gesamt-Dateien > 200 MB warnen ("Das kann bei aelteren Geraeten langsam sein")
- Per-Datei-Limit als UI-Validation konfigurierbar (CONVENTIONS.md Default: kein Limit, aber Warnung)
