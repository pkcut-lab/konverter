# Dossier — pdf-zu-jpg

## § 1 — Tool-Mechanik-Research

### Kern-Technologie: PDF.js Client-Side Rendering

PDF ist ein vektorbasiertes Format, das intern Seiten als Kombination aus Vektorgrafiken, eingebetteten Bitmaps, Schriften und Metadaten beschreibt. Für die Konvertierung zu JPG muss jede Seite **rasterisiert** werden — d.h. auf ein Pixel-Grid projiziert werden.

**Client-Side-Pfad via PDF.js (MIT-Lizenz) + Canvas API:**
```
PDF-Datei → ArrayBuffer (FileReader)
→ pdfjsLib.getDocument() → PDFDocumentProxy
→ pdfDocument.getPage(n) → PDFPageProxy
→ page.getViewport({ scale: targetDPI / 72 })
→ page.render({ canvasContext, viewport })
→ canvas.toBlob('image/jpeg', quality)   // Blob für direkten Download
→ URL.createObjectURL(blob)              // kein Server-Upload nötig
```

**DPI-Mechanik:**
- PDFs verwenden intern 72 Punkte/Zoll (pt/inch) als Basiseinheit.
- Zur Konvertierung auf Ziel-DPI: `scale = desiredDPI / 72`
- 72 DPI → scale = 1.0 (Web-minimal)
- 150 DPI → scale ≈ 2.08 (Standard Bildschirm/Screen)
- 300 DPI → scale ≈ 4.17 (Druck-Qualität)
- Bei 300 DPI ist ein DIN-A4-Seite (21×29.7 cm) = 2480×3508 Pixel → ~26 MB unkomprimiert

**JPEG-Qualitätsparameter:**
- `canvas.toBlob('image/jpeg', quality)` nimmt quality 0.0–1.0
- Empfohlen: 0.92 als Default (visuell verlustfrei für Dokumente), 0.75 für kompakte Ausgabe
- Kein Qualitätsverlust bei Vektorgrafiken/Text gegenüber höherem scale, solange JPEG-Quality ≥ 0.85

**Sprach-Varianten:**
- Primärbezeichnung DE: "PDF in JPG umwandeln" / "PDF zu JPG konvertieren"
- Alternativen: "PDF als Bild speichern", "PDF Seiten als JPG exportieren", "PDF in Bild umwandeln"
- Technische Variante: "PDF rasterisieren"

**Besondere Dokumenttypen (Edge-Cases):**
- Passwortgeschützte PDFs: PDF.js löst `PasswordException` → Passwort-Prompt als Dialog
- Scanned PDFs: Enthält nur eingebettete Bitmaps; hochskalieren bringt keine Schärfe
- PDFs mit transparenten Ebenen: Canvas-Hintergrund = Weiß bei JPG (kein Alpha-Kanal in JPEG)
- Mehrseitige PDFs: Jede Seite = ein JPG; ZIP-Download als Batch-Output
- Sehr große PDFs (>30 Seiten, 300 DPI): Canvas-Speicher pro Seite ~26 MB → Warnung und seitenweise Verarbeitung statt Bulk

**Präzisions-Erwartungen:**
- Privat-Nutzer (Scan für E-Mail): 72–150 DPI ausreichend
- Professionell (Print-Vorstufe, Präsentation): 300 DPI erforderlich
- Standard-Preset: 150 DPI (Balance aus Qualität und Dateigröße)

## § 5 — UX-Patterns Best-in-Class

### Beobachtete Patterns (aus Konkurrenz-Analyse)

- **DPI-Vorauswahl als RadioGroup:** pdf2jpg.net und freepdfconvert.com nutzen 3 Stufen mit klaren Labels ("Average / Good / Excellent" oder "72/150/300"). Reduziert kognitive Last gegenüber freiem Slider.
- **DPI-Slider + Qualitätsprozentsatz:** PDF24 bietet die mächtigste Konfiguration, aber höchste kognitive Komplexität.
- **"Normal / High"-Labels:** iLovePDF vereinfacht maximal — gut für Consumer, schlecht für Präzisions-Nutzer.
- **Seiten-Auswahl:** Nur sejda (Quick-Save-Button pro Seite) und Adobe (Checkbox-Seitenauswahl) bieten das. Direkt relevanter UX-Vorteil für Nutzer, die nur Seite 1 oder Seiten 3–5 brauchen.
- **Download-Pattern:** ZIP-Archiv für Multi-Page ist Standard. Einzel-Download pro Seite zusätzlich ist Better-Practice.
- **Fortschrittsanzeige:** Alle Server-Konverter zeigen Spinner/Progress-Bar — bei client-seitigem Rendering noch wichtiger (Canvas-Render dauert messbar).
- **Privacy-Badge:** "Dateien werden nach X Stunden gelöscht" bei pdf2jpg.net, sejda. Bei client-seitigem Tool: "Ihr PDF verlässt Ihren Browser nie" wäre stärkeres Signal.

### Empfehlung

- **Übernehmen:** DPI-RadioGroup mit 3 klaren Stufen (72/150/300), Seiten-Vorschau mit Checkbox-Selektion, ZIP + Einzeldownload, Fortschrittsbalken pro Seite
- **Besser machen:** "Kein Upload" als primäres Hero-Signal (adressiert §4 Privacy-Pain 1+2+4), Seiten-Auswahl per Vorschau-Grid (adressiert fehlendes Feature in 7/9 Konkurrenten)
- **Bewusst weglassen:** Cloud-Import (Drive/Dropbox) — würde Server-Seitenabhängigkeit erzwingen, widerspricht Non-Negotiable

- 1-seitige PDF → Single JPG, kein ZIP nötig
- 100-seitige PDF @ 300 DPI → Warnung, seitenweise Verarbeitung mit Pause

- PDF mit sehr dunklem Scan → bei 72 DPI lesbar, bei 300 DPI ideal
- Passwort-geschützte PDF → Prompt-Dialog, bei falschem Passwort klare Fehlermeldung

## § 9 — Differenzierungs-Hypothesen

### H1 — Client-Side als einziger echter Datenschutz-Beweis im DE-Markt

**These:** Kittokit ist der einzige deutsche PDF-zu-JPG-Konverter, der technisch beweisbar keinen Server-Upload durchführt — via PDF.js direkt im Browser. "Ihr PDF verlässt Ihren Browser nie" ist nicht nur ein Versprechen, sondern eine technische Eigenschaft.

**Beleg:** §2 Matrix zeigt 9/9 Konkurrenten mit Server-Upload-Pflicht. §4 Pain 1+2 zeigen, dass Privacy der stärkste ungelöste Pain für dieses Tool-Segment ist — verstärkt durch KI-Training-Sorgen ab 2025. §7 Keyword-Gap: `pdf jpg konverter ohne upload` unbesetzt im DE-Markt.

**Wert-Beleg:** Professionelle Nutzer (Juristen, Steuerberater, HR) konvertieren täglich sensible Dokumente. Ein glaubwürdiges "kein Upload" schlägt jedes "wir löschen nach 2h"-Versprechen.

---

### H2 — Seiten-Auswahl per visuelles Vorschau-Grid

**These:** Nur 2 von 9 Konkurrenten bieten irgendeine Form von Seitenauswahl. Ein Vorschau-Grid aller PDF-Seiten mit Checkbox-Selektion ("Seiten 1, 3 und 5 auswählen") löst den häufigsten Use-Case — einzelne Seiten aus einem längeren Dokument extrahieren — besser als alle aktuellen Lösungen.

**Beleg:** §4 Pain (Missing-Feature) — kein direktes Zitat, aber §2 zeigt, dass 7/9 Konkurrenten keine Seitenauswahl bieten. §5 People-Also-Ask: "Kann ich eine einzelne Seite aus einem PDF als JPG speichern?" bestätigt diesen Intent. §3 sejda Weakness: Quick-Save pro Seite ist Behelfslösung, kein echtes Selektion-Interface.

**Wert-Beleg:** Für Use-Case "nur Deckblatt als JPG" oder "Seiten 2–4 für Präsentation" ist dies die häufigste manuelle Aufgabe — aktuell bei 7/9 Konkurrenten nur durch "gesamtes PDF konvertieren und dann löschen" lösbar.

---

### H3 — 300 DPI kostenlos + Visueller DPI-Vergleich

**These:** 300 DPI ist bei den meisten Konkurrenten hinter der Paywall. Kittokit bietet es kostenlos an (client-seitig ist DPI-Berechnung free) und erklärt den Unterschied mit einem visuellen Before/After-Sample — einziger Informations-Gain-Kandidat in dieser Nische.

**Beleg:** §2 Matrix: nur pdf2jpg.net bietet 300 DPI kostenlos; freepdfconvert.com sperrt 300 DPI hinter Pro. §6 Keyword-Gap: `pdf zu jpg hohe qualität` unbesetzt. §7 unterrepräsentiert: kein Konkurrent zeigt DPI-Vergleichsvisualisierung.

## § 11 — Security-Guidance (tool-type-abhängig)

**tool_type = `converter` (PDF → Image)**

PDF-zu-JPG ist kein klassischer Security-Risikokonverter (kein Regex-Input, kein User-Code-Execution), aber der PDF-Parser birgt spezifische Risiken:

- **Threat-Model:** Malformed oder malicious PDFs können PDF.js-Parsing-Fehler auslösen. PDF.js wurde designed um im Browser-Sandbox zu laufen — das limitiert Blast-Radius erheblich, aber `getDocument()`-Fehler müssen abgefangen werden.
- **Pflicht-Pattern:**
  ```javascript
  try {
    const pdf = await pdfjsLib.getDocument({ data: arrayBuffer }).promise;
    // render pages
  } catch (err) {
    if (err.name === 'PasswordException') {
      showPasswordPrompt();
    } else if (err.name === 'InvalidPDFException') {
      showError('Diese PDF-Datei ist beschädigt oder ungültig.');
    } else {
      showError('PDF konnte nicht geladen werden.');
    }
  }
  ```
- **Verboten:**
  - `eval()` auf PDF-Inhalte oder Dateinamen
  - Direkte `innerHTML`-Injection mit Dateinamen (XSS-Gefahr)
