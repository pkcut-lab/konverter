# Dossier — pdf-komprimieren

## § 1 — Tool-Mechanik-Research

### Was PDF-Komprimierung technisch macht

PDF-Komprimierung reduziert Dateigroesse durch eine oder mehrere dieser Methoden, je nach Ansatz:

**Verlustbehaftet (Lossy) — groesste Reduktionsraten, 50–85%:**
- **Bild-DPI-Reduktion**: Eingebettete Bilder werden von 300 DPI auf 72/150 DPI runterskaliert. Formel: `scale = targetDPI / sourceDPI`. Fuer Web optimal: 72–96 DPI. Fuer Druck: 150–300 DPI.
- **JPEG-Qualitaets-Re-Encoding**: Bilder werden mit niedrigerer JPEG-Qualitaetsstufe (z.B. 60–75%) neu kodiert. Sichtbare Qualitaetsverluste erst unter 50%.
- **Graustufenkonvertierung**: Farb-RGB nach Graustufen. Weitere ~30% Reduktion moeglich.

**Verlustlos (Lossless) — moderate Reduktionsraten, 5–25%:**
- **Flate/Deflate-Optimierung**: Alle Content-Streams werden mit maximaler Deflate-Kompression neu kodiert.
- **Duplikat-Stream-Deduplication**: Identische Objekte im PDF (z.B. Schriftdaten, Formulare) werden zusammengefuehrt.
- **Metadata-Entfernung**: XMP-Metadaten, EXIF, Kommentare, Formularfeld-Daten.
- **Font-Subsetting**: Eingebettete Schriften auf tatsaechlich verwendete Glyphen reduzieren.
- **Objekt-Streams**: PDF-1.5+ ermoeglicht cross-reference-streams statt xref-Tables — spart 10–30 KB bei mittelgrossen Dateien.

**Technische Basis-Formeln:**
- `compressionRatio = (originalSize - compressedSize) / originalSize * 100`
- Beispiel-Erwartungswerte fuer ein typisches 5 MB Scan-PDF: Extreme (~85%), Standard (~60%), Lossless (~15%)

### Typische Eingabe-Typen und erwartete Kompressionsraten

| PDF-Typ | Dominante Methode | Erwartete Reduktion |
|---|---|---|
| Scanned-Dokument (Bitmap-only) | Bild-DPI + JPEG-Re-Encoding | 60–85% |
| Praesentation (viele Bilder) | Bild-DPI | 50–75% |
| Text-PDF (kein Bild) | Flate + Font-Subset | 5–20% |
| Formulare (interaktiv) | Formular-Flatten + Metadata | 10–30% |
| Gemischtes Dokument | Kombination | 30–65% |

### Praezisions-Erwartungen der Zielgruppe

- **Web-Upload-Nutzer** (E-Mail-Anhang, Portale): wollen < 10 MB, oft < 5 MB — Reduktionsrate egal, Ergebnis zaehlt
- **DSGVO-bewusste Nutzer** (Geschaeftsdokumente): wollen keinen Server-Upload
- **Print-Profis** (Agentur, Druck): 150–300 DPI Minimum — verlustlose Option bevorzugt

### Haeufige Nutzerfehler
- PDF mit eingebetteten Bildern mit Text-PDF verwechseln — Erwartung zu hoch
- "Ohne Qualitaetsverlust" Erwartung bei stark bild-haltigem PDF — erfordert ehrliches UX-Feedback

## § 3 — Strengths / Weaknesses Top 3

### Konkurrent 1 — tools.pdf24.org/de/pdf-komprimieren [Quelle: 3]

**Strengths:**
- Groesszuegigster Free-Tier: 500 MB, unlimitierte Aufgaben, kein Rate-Limit
- Server in Deutschland — starkste Privacy-Botschaft unter kostenlos-werbefinanzierten Tools
- DPI-Slider + Qualitaets-Prozent geben Nutzern maximale Kontrolle

**Weaknesses:**
- Ads direkt im Tool-UI — stoert Workflow
- Trotz DE-Server: weiterhin vollstaendiger Datei-Upload — kein "zero knowledge"
- Kein Before/After-Groessen-Feedback vor Download

### Konkurrent 2 — ilovepdf.com/de/pdf_komprimieren [Quelle: 1]

**Strengths:**
- Bekannteste Marke im DE-Markt fuer PDF-Tools
- 3-Level-Auswahl (Extreme/Recommended/Lower) ist intuitiv und reduziert Entscheidungsfrust
- Cloud-Integration (GDrive, Dropbox) fuer Cloud-Nutzer praktisch

**Weaknesses:**
- Hoechste Tracking-Dichte: 143 TCF-Vendors + 62 Werbepartner — massiver Privacy-Fail
- 200 MB / 2 Dateien gratis — schnell erschoepft fuer Batch-Komprimierung
- Keine Transparent-Anzeige der erwarteten Reduktionsrate

### Konkurrent 3 — sejda.com/compress-pdf [Quelle: 5]

**Strengths:**
- Detaillierteste Optionen: JPEG-Qualitaet 65/72/85%, DPI 72–720, Graustufen
- Klarste Privacy-Kommunikation ("Files stay private. Automatically deleted after 2 hours.")
- Sauberste UI ohne Overload von Options in der Haupt-Ansicht

**Weaknesses:**
- Striktestes Rate-Limit: 3 Tasks/Stunde — frustrierend fuer Batch-Komprimierung
- EN-only — schwache DACH-Ansprache
- 100 MB / 50 Seiten Grenze schliesst grosse Dokumente aus

## § 5 — UX-Patterns Best-in-Class

*Erbt Basis aus Category-Root §5. Spezifiziert fuer Komprimierungs-Tool.*

### Komprimierungs-Level-Auswahl

- **Label-basiert (ilovepdf-Muster)**: "Extreme / Recommended / Lower" — niedrigstes Entscheidungs-Gewicht, beste Conversion bei Mainstream-Nutzern
- **DPI/Prozent-basiert (pdf24/sejda-Muster)**: Maximale Kontrolle fuer Power-User — aber hoeheres kognitive Last
- **Empfehlung fuer kittokit**: Zwei Ebenen: sichtbare Label-Wahl (3 Stufen: Standard / Maximal / Verlustlos) + erweiterbare "Experten-Optionen" (DPI, Qualitaets-Prozent) — trifft beide Zielgruppen

### Feedback-Loop (White-Space-Opportunity)

Kein einziger Konkurrent zeigt "Voraussichtliche Reduktion: ~60%" vor dem eigentlichen Download. Moegliches Pattern:
```
[ Datei hochgeladen: 8,4 MB ]
Komprimierungsstufe: [ Standard v ]

Voraussichtliche Groesse: ~3,2 MB (−62%)  ← Schaetzung basierend auf PDF-Typ
[ PDF komprimieren ]
```

Nach Komprimierung:
```
Ergebnis: 3,1 MB (−63%)   Original: 8,4 MB
[ Herunterladen ]  [ Nochmal mit anderer Stufe ]
```

### Error-Feedback
- Passwortgeschuetztes PDF: Inline-Passwort-Prompt (kein Page-Reload)
- Nicht-PDF hochgeladen: Shake-Animation + Meldung "Nur PDF-Dateien werden unterstuetzt"
- PDF bereits komprimiert: Hinweis-Banner "Diese PDF ist bereits gut optimiert (geschaetzte Reduktion < 5%)"

### Mobile-Tastatur
- Kein nummerisches Input noetig fuer Haupt-Flow — Tastatur-Typ irrelevant
- Upload via native File-Picker auf Mobile gut unterstuetzt

### Reverse-Toggle / Stufen-Switch
- "Nochmal mit anderer Stufe komprimieren" Button nach Download — verhindert kompletten Neu-Upload

## § 8 — Edge-Cases + Validation-Regeln

### Input-Validierung
| Input-Klasse | Status | Fehlermeldung (DE) |
|---|---|---|
| Nicht-PDF-Datei (.jpg, .docx, etc.) | Blockiert | "Bitte eine PDF-Datei hochladen (.pdf)" |
| Passwortgeschuetztes PDF | Erlaubt mit Prompt | "Diese PDF ist passwortgeschuetzt. Bitte Passwort eingeben:" |
| PDF-Dateigroe 0 Bytes | Blockiert | "Die Datei scheint leer zu sein. Bitte erneut versuchen." |
| Korrumpiertes / ungueltiges PDF | Blockiert | "Diese Datei konnte nicht als PDF gelesen werden. Moegliche Ursache: Datei ist beschaedigt." |
| Sehr kleines PDF (< 50 KB) | Warnung | "Diese PDF ist bereits sehr klein. Komprimierung bringt moeglicherweise wenig Ergebnis." |
| Sehr grosse PDF (> 20 MB client-side) | Warnung je nach Architektur | "Grosse Dateien benoetigen laenger. Bitte Seite offen lassen." |

### Boundary-Tests
- 0 Bytes: Blockieren
- 1 Byte: Blockieren (kein gueltiges PDF)
- 1 KB: Warnen (kein sinnvoller Komprimierungsgewinn)
- 50 MB: Verarbeiten (je nach Architektur, ggf. Warnung)
- 500 MB: Browser-Memory-Grenze, Warnung oder Ablehnen bei client-seitiger Verarbeitung

### PDF-Typ-Erkennungs-Heuristik (fuer Schaetzung)
```
if (imageBytes / totalBytes > 0.7) → "Bild-dominiertes PDF" → hohe Reduktionsrate moeglich (50–80%)
if (imageBytes / totalBytes < 0.2) → "Text-PDF" → geringe Reduktionsrate (5–20%), ehrlich kommunizieren
```

## § 9 — Differenzierungs-Hypothesen

### H1 — Before/After Groessen-Schaetzung (kein Konkurrent macht das)

**These:** Wenn kittokit als einziger Konkurrent eine *geschaetzte* Endgroesse vor dem Download anzeigt ("Voraussichtliche Groesse: ~3,2 MB, −62%"), loest das den staerksten User-Pain (§4, Missing-Feature-Pain) und schafft ein messbares UX-Alleinstellungsmerkmal.

**Beleg:** §2 Matrix: 10/10 Konkurrenten zeigen keine Before/After-Schaetzung. §4 Pain 3: "I never know if it actually worked until I download it." Technisch realisierbar: PDF-Bild-Byte-Anteil analysieren, Schaetzwert mit +-Toleranz ausgeben (kein echter Komprimierungs-Run noetig fuer Schaetzung).

**Umsetzungs-Hinweis:** Schaetzung als Schaetzung labeln ("ungefaehr ~3 MB") — keine harte Garantie. Vermeidet falschen Qualitaetsversprechen-Vorwurf.

### H2 — Verlustlose Komprimierung prominent anbieten (kein Konkurrent positioniert das als Feature)

**These:** Die "Lossless"-Option (Flate-Optimierung, Font-Subsetting, Metadata-Entfernung) ohne jeglichen Qualitaetsverlust ist in jedem Konkurrenten versteckt oder fehlt. Wer sie als explizites, prominentes Feature benennt ("Verlustlos — kein Pixel-Verlust, typisch 5–20%"), trifft Privacy-/Qualitaets-Nutzer die Dokumente fuer Print oder Archiv komprimieren.

**Beleg:** §2 Matrix: pdf24 und sejda bieten Lossless an, aber ohne Feature-Positionierung. §4 Pain 4: "There's no way to know what you'll get before you click" — Lossless loest Quality-Uncertainty.

### H3 — Ehrliches PDF-Typ-Feedback (kein Konkurrent tut das)

**These:** Wenn kittokit dem Nutzer erklaert "Diese PDF ist text-dominiert — Komprimierung bringt typisch nur 5–15%" (statt alle PDF-Komprimierungen gleich zu behandeln), baut das signifikant mehr Vertrauen auf als die Standard-"Sicher-und-schnell"-Botschaft.

**Beleg:** §7 Content-Angle: keine Konkurrenz erklaert PDF-Typ-spezifische Komprimierungs-Erwartungen. §1 Tool-Mechanik-Research: Text-PDFs haben fundamental andere Komprimierungs-Charakteristika als Bild-PDFs — dies ist valides technisches Wissen das Nutzer benotigen.

## § 11 — Architecture-Guidance (Architektur-Entscheidung noetig)

> **Kritischer Befund:** PDF-Komprimierung faellt in eine Kategorie, die einen direkten Konflikt mit dem kittokit Non-Negotiable §2 ("kein Server-Upload, Ausnahme 7a ML-File-Tools") verursacht.

### Threat-Model / Architektur-Konflikt

PDF-Komprimierung (besonders Lossy: Bild-Re-Encoding) erfordert eine vollstaendige PDF-Parsing + Bild-Resampling-Engine. Alle verfuegbaren, serioesen Loesungen sind entweder:
- **Server-seitig** (Ghostscript, Poppler, MuPDF) — verletzt Non-Negotiable §2
- **WASM-basiert mit AGPL-Lizenz** (MuPDF.wasm, Ghostscript WASM, ps-wasm) — verletzt MIT-Lizenz-Constraint
- **Pure-JS / MIT-lizenziert** (pdf-lib, pdfmake, jsPDF) — bieten **keine** Bild-Komprimierungs-APIs

### Optionen fuer den Tool-Builder

**Option A: Lossless-only Client-Side via pdf-lib (MIT)**
- Implementierung: pdf-lib (MIT) fuer Strukturoptimierung (Flate-Recompression, Metadata-Strip, Font-Subset)
- Erwartete Reduktionsrate: 5–25% (nur strukturelle Optimierung)
- Privacy: 100% client-side, kein Upload
- Qualitaetsversprechen: "Verlustlose Optimierung" — ehrlich und belegbar
- Trade-off: Nutzer die 60–80% Reduktion erwarten werden enttaeuscht — ehrliche Kommunikation der Grenzen ist Pflicht
- **Empfehlung: Geeignet wenn Privacy absolut Vorrang hat. Positionierung als "verlustloser Optimierer" statt "Komprimierer".**

**Option B: AGPL-WASM Bibliothek (MuPDF.wasm, Ghostscript WASM)**
- Implementierung: Vollstaendige PDF-Komprimierung inkl. Bild-Re-Encoding, client-side
- Erwartete Reduktionsrate: 50–85%
- Privacy: 100% client-side, kein Upload
- Lizenz-Risiko: AGPL-3.0 — wenn kittokit kein open-source Projekt ist, erfordert AGPL eine kommerzielle Lizenz von Artifex (MuPDF) oder ist inkompatibel (Ghostscript)
- Kosten: Artifex-Kommerziallizenz nicht oeffentlich — Preisanfrage noetig
- **Empfehlung: Nicht autonom entscheiden — User-Approval via CEO-Escalation erforderlich.**

**Option C: Server-seitige Verarbeitung als "File-Processing-Ausnahme"**
- Analog zu 7a ML-File-Tools: Server-Upload mit sofortiger Loeschung, kein Speichern
- Implementierung: Ghostscript oder MuPDF auf eigenem Server (Cloudflare Workers mit WASM, oder separater Service)
- Erwartete Reduktionsrate: 50–85%
- Privacy: "Sofort geloescht nach Verarbeitung" — analog zu pdf24, pdfforge
- Non-Negotiable-Status: Verletzt §2 NICHT wenn explizit als "File-Processing-Ausnahme" von User genehmigt
- **Empfehlung: Pragmatischste Option fuer Nutzerwert. Erfordert explizite User-Genehmigung analog zu ML-Tool-Ausnahme.**

### Pflicht-Pattern fuer Builder

**MINIMUM (autonom umsetzbar ohne User-Genehmigung):**
```
Option A: pdf-lib (MIT) Lossless-Optimierung, client-side
- Positionierung: "PDF verlustlos optimieren (5–25% typisch)"
- Kein Upload, kein Server
- Klare Kommunikation der Grenzen
```

**FUER VOLLSTAENDIGE KOMPRIMIERUNG (erfordert User-Entscheidung):**
