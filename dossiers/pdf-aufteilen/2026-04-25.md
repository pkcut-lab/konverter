## § 1 — Tool-Mechanik-Research

### Kern-Operation

PDF-Splitting bedeutet: eine Menge von Seiten aus einer Quell-PDF-Datei in eine oder mehrere neue
PDF-Dateien schreiben. Die Seiten-IDs (0-basierter Array-Index) sind der zentrale Parameter.

**Technisch mit pdf-lib (MIT-Lizenz):**

```javascript
import { PDFDocument } from 'pdf-lib';

async function splitByRange(arrayBuffer, ranges) {
  // ranges = [[0, 2], [3, 6]] (0-basiert, end inklusiv)
  const srcDoc = await PDFDocument.load(arrayBuffer); // kann PasswordException werfen
  const results = [];
  for (const [start, end] of ranges) {
    const destDoc = await PDFDocument.create();
    const pageIndices = Array.from({ length: end - start + 1 }, (_, i) => start + i);
    const pages = await destDoc.copyPages(srcDoc, pageIndices);
    pages.forEach(p => destDoc.addPage(p));
    results.push(await destDoc.save());
  }
  return results; // Uint8Array[] — je eine Datei
}
```

Quelle: [npm/pdf-lib Dokumentation, fetched 2026-04-25], zitiert in §1

### Split-Modi (Implementierungskandidaten)

| Modus | Beschreibung | Beispiel | Komplexität |
|---|---|---|---|
| **Seitenbereich (Range)** | Nutzer gibt "1-3, 5, 7-9" an | 12-seitige PDF → 3 Ausgabe-PDFs | Mittel |
| **Feste Seitenanzahl** | Alle N Seiten trennen | 12 Seiten / 4 Seiten → 3 PDFs | Niedrig |
| **Jede Seite einzeln** | Jede Seite wird eigene Datei | 12 Seiten → 12 PDFs | Niedrig |
| **Gerade/Ungerade** | Seiten 2,4,6 vs. 1,3,5 | Für Duplexdruck-Workflows | Niedrig |

Baseline-Pflicht (alle 6 Konkurrenten): Seitenbereich-Modus + feste Seitenanzahl.
Differenzierungs-Kandidat: Live-Vorschau im Browser ohne Upload (client-side + PDF.js).

### Berechnung: Seitenbereich-Parsing

Eingabe "1-3, 5, 7-9" → Normalisierung:
1. Split auf Komma → `["1-3", "5", "7-9"]`
2. Für jeden Token: Split auf Bindestrich
   - "1-3" → `[0, 2]` (0-basiert, internen Index = input − 1)
   - "5" → `[4, 4]`
   - "7-9" → `[6, 8]`
3. Validierung: alle Indizes < `srcDoc.getPageCount()`, keine negativen Zahlen, start ≤ end

### Präzisions-Erwartungen

PDF-Splitting ist verlustlos — Seiten werden 1:1 kopiert, keine Re-Kodierung.
Ausnahme: eingebettete Formularfelder — pdf-lib kopiert Felder nicht, nur die gerenderte Seite.
Wenn ein PDF interaktive Formulare enthält, ist das ein bekanntes pdf-lib-Limit.

### Häufige Nutzer-Fehler

- Eingabe "0" als erste Seite (PDF-Nutzer denken 1-basiert, Implementierung ist 0-basiert intern)
- Bereich "5-3" (umgekehrte Range) → explizite Fehlermeldung nötig
- Leerzeichen in der Range-Eingabe → robustes Trimming
- Komma am Ende ("1-3,") → graceful Ignore letzten leeren Token

## § 5 — UX-Patterns Best-in-Class

*Basis aus parent_dossier §5 geerbt. Nachfolgend Split-spezifische Ergänzungen.*

### Split-spezifische UX-Patterns

**Schere-UX (Smallpdf, PDF2Go):**
- Jede Seiten-Grenze hat ein klickbares Schere-Icon
- Klick = gestrichelte Trennlinie erscheint
- Intuitiv für visuelle Nutzer, aber schlecht für Power-User ("Gib mir Seiten 1-50" als Klick-Arbeit)

**Range-Text-Input (iLovePDF):**
- Freitextfeld: "1-3, 5, 7-9"
- Schnell für Power-User, aber nicht intuitiv für Gelegenheitsnutzer
- Kombination mit Thumbnail-Vorschau ist der Sweet-Spot

**Split-Mode-Dropdown (PDF24):**
- Dropdown mit 4 vordefinierten Modi + Parameterfeld (z.B. "alle N Seiten")
- Reduziert Entscheidungsaufwand, verbirgt aber Komplexität

**Empfehlung für kittokit:**
Zwei-Modi-UI: (1) **Visuell** — Thumbnail-Grid, Klick auf Seiten-Grenzen; (2) **Schnell** —
Range-Text-Eingabe "1-3, 5, 7-9". Beide Modi aktivieren sofort die client-seitige Verarbeitung
ohne Upload-Delay.

**Mobile-UX:**
- `inputmode="numeric"` für Seitenzahl-Inputs
- Touch-freundliche Schere-Targets (mindestens 44px × 44px)
- Thumbnail-Grid als horizontaler Scroll auf Mobile

**Status-Feedback:**
- Verarbeitungs-Progress: da client-side fast instant bei normalen Dokumenten, kann auf
  Spinner verzichtet werden — stattdessen direkter Download-Button erscheint

## § 8 — Edge-Cases + Validation-Regeln

**Allowed:**
- Ganzzahlen ≥ 1 in Range-Eingabe (intern: − 1 für 0-basierte Indizes)
- Komma-separierte Ranges: "1-3, 5, 7-9" (mit/ohne Leerzeichen)
- Einzelseiten als Shorthand für Range: "5" = "5-5"
- Vollständige PDF (Range = alle Seiten) → Identische Ausgabe-Datei, kein Fehler

**Blocked → Fehlermeldung:**

| Input | Fehlermeldung (DE) |
|---|---|
| Seitenzahl > Gesamtseiten | "Seite {n} existiert nicht. Das PDF hat nur {max} Seiten." |
| Umgekehrte Range "5-3" | "Ungültiger Bereich: Die Startseite muss kleiner als die Endseite sein." |
| Seitenzahl 0 oder negativ | "Seitenzahlen beginnen bei 1." |
| Leere Auswahl | "Bitte mindestens eine Seite auswählen." |
| Nicht-PDF-Datei | "Bitte eine PDF-Datei hochladen." |
| Passwortgeschützte PDF | "Diese PDF ist passwortgeschützt. Bitte geben Sie das Passwort ein oder verwenden Sie eine ungeschützte PDF." |
| Beschädigte/korrupte PDF | "Die PDF-Datei ist beschädigt und kann nicht verarbeitet werden." |
| Zu große Datei (>200 MB) | "Diese Datei ist sehr groß ({size} MB). Die Verarbeitung kann einige Sekunden dauern." (Warnung, kein Block) |

**Boundary-Tests:**
- Erste Seite (1): ✓
- Letzte Seite (N): ✓
- Einseitige PDF: Ausgabe = identische 1-Seiten-PDF; Hinweis "Diese PDF hat nur eine Seite"
- 0-Input: → Fehler "Seitenzahlen beginnen bei 1"
- Negativer Input: → Fehler
- Sehr lange Range-Eingabe (>500 Zeichen): → Kürzungs-Guard
- Leeres Komma am Ende ("1-3,"): → Letzten leeren Token ignorieren

## § 9 — Differenzierungs-Hypothesen

**H1: Client-Side (kein Upload)** — Beleg: §4 Privacy-Pain (3 Zitate) + §2 White-Space-Analyse

Kein einziger der 6 Hauptkonkurrenten bietet im Web-Tool rein client-seitige Verarbeitung an.
pdf-lib (MIT) ermöglicht vollständiges PDF-Splitting ohne Server-Kontakt. Die Formulierung
"Ihre Datei verlässt Ihren Browser nie" ist faktisch belegbar und adressiert den stärksten
belegten User-Pain in der Kategorie. Besonders relevant für DACH: DSGVO-bewusste Nutzer und
Unternehmensrichtlinien gegen Dokument-Upload auf Drittserver.

Investition: ~10–15h Implementierungsaufwand für pdf-lib-Integration + UI.
Risiko: Browser-RAM-Limit bei sehr großen PDFs (>200 MB) — Warnung statt Block. File API (FileReader)
ist universell unterstützt (alle modernen Browser, kein Polyfill nötig).

**H2: Keine künstlichen Limits** — Beleg: §2 Matrix (Limit-Spalte) + §4 Limit-Pain

Alle 6 Konkurrenten haben business-motivierte Limits (Server-Kosten → Upsell). Client-side
hat keine Server-Kosten → keine Motivation für Limits. Das einzige reale Limit ist Browser-RAM.
Formulierung: "Kein Dateilimit — so groß wie Ihr Browser es verarbeiten kann."

Caveat: Sehr große PDFs (>200 MB) können den Browser-Tab zum Absturz bringen. Warnung
bei >100 MB statt hartem Block.

## § 11 — Security-Guidance

*Trigger: Keywords "parse", "password" im Dossier-Scope (AGENTS.md Schritt D2).*

**Threat-Model:** pdf-lib lädt beliebige Binärdateien (ArrayBuffer) — malformed oder corrupt PDFs
können unkontrollierte Exceptions werfen. Passwortgeschützte PDFs haben separaten
PasswordException-Pfad. Keine ReDoS-Gefahr (kein User-Regex-Input). Kein Prototype-Pollution-Risiko
(keine JSON.parse-Varianten mit User-Merge).

**Pflicht-Pattern:**

```javascript
import { PDFDocument, PasswordException } from 'pdf-lib';

async function loadSafe(arrayBuffer) {
  try {
    return await PDFDocument.load(arrayBuffer);
  } catch (err) {
    if (err instanceof PasswordException) {
      throw new Error('PASSWORD_PROTECTED');
    }
    throw new Error('CORRUPT_PDF');
  }
}
```

Zusätzlich: Größen-Check vor dem Laden:

```javascript
if (file.size > 200 * 1024 * 1024) {
  showWarning(`Diese Datei ist ${(file.size / 1e6).toFixed(0)} MB groß. Die Verarbeitung kann etwas dauern.`);
}
```

**Verboten:**
- `eval()` auf PDF-Inhalt (nicht anwendbar für pdf-lib — PDF.js hat ähnliche Einschränkungen)
- Ungeprüfte `JSON.parse()` auf Nutzer-Input (kein JSON in diesem Tool)
- Roher Zugriff auf `arrayBuffer` ohne try/catch-Wrapper

**Quellen:**
- MDN File API: https://developer.mozilla.org/en-US/docs/Web/API/File_API — gefetcht 2026-04-25
- OWASP: https://owasp.org/www-community/attacks/Prototype_pollution — nicht direkt relevant,
  zur Vollständigkeit geprüft
